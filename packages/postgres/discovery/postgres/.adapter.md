# Postgres Storage Bridge Adapter

Postgres maps LISTEN/NOTIFY bridge trigger payloads into Relayfile StorageBridgeEvent envelopes under /postgres/{db}/{schema}/{table}/{pk}.json. The adapter validates configuration, normalizes push notifications, publishes storage events, fetches content through injected provider clients, and resolves file-native writeback into provider-shaped operations.

## Read-only Mounts

Read-only synchronized content is mounted at `/postgres/{db}/{schema}/{table}/{pk}.json`. Deleted provider objects emit deletion events with null fetch content.

## Writable Resources

The resource `/postgres/{db}/{schema}/{table}` ships `/postgres/{db}/{schema}/{table}/.schema.json` and `/postgres/{db}/{schema}/{table}/.create.example.json`.

The resource `/postgres/listeners` ships `/postgres/listeners/.schema.json` and `/postgres/listeners/.create.example.json`.

The static discovery resource `/postgres/rows/.schema.json` ships `/postgres/rows/.create.example.json`.

## Operations

Create writes use any non-canonical filename in the resource directory. Canonical filenames patch mutable fields only. Removing a canonical file resolves to provider delete operations. Read-only fields such as `id`, `createdAt`, `updatedAt`, `url`, `_webhook`, and `_connection` are rejected before provider calls.

The object resource resolves to `parameterized INSERT/UPDATE/DELETE`. The lifecycle resource resolves to `LISTEN {channel}`.

## Push and Auth Validation

Database notifications are trusted through the configured database connection. Webhook-style local fixtures can use x-relayfile-signature HMAC.

## Nango template

No Nango anchor; spec source: PostgreSQL LISTEN/NOTIFY trigger payloads and parameterized row writes.

## Nango Fallback

Postgres is not in the Nango scheduled fallback catalog for this wave; operators must rely on the real-time push path.

## ID Patterns

`/postgres/{db}/{schema}/{table}`: `^.+$`.

`/postgres/listeners`: `^[A-Za-z_][A-Za-z0-9_]*$`.
