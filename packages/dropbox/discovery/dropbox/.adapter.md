# Dropbox Storage Bridge Adapter

Dropbox maps Dropbox webhooks plus files/list_folder/continue cursors into Relayfile StorageBridgeEvent envelopes under /dropbox/{accountId}/{path}. The adapter validates configuration, normalizes push notifications, publishes storage events, fetches content through injected provider clients, and resolves file-native writeback into provider-shaped operations.

## Read-only Mounts

Read-only synchronized content is mounted at `/dropbox/{accountId}/{path}`. Deleted provider objects emit deletion events with null fetch content.

## Writable Resources

The resource `/dropbox/{accountId}/files` ships `/dropbox/{accountId}/files/.schema.json` and `/dropbox/{accountId}/files/.create.example.json`.

The resource `/dropbox/cursors` ships `/dropbox/cursors/.schema.json` and `/dropbox/cursors/.create.example.json`.

The static discovery resource `/dropbox/files/.schema.json` ships `/dropbox/files/.create.example.json`.

## Operations

Create writes use any non-canonical filename in the resource directory. Canonical filenames patch mutable fields only. Removing a canonical file resolves to provider delete operations. Read-only fields such as `id`, `createdAt`, `updatedAt`, `url`, `_webhook`, and `_connection` are rejected before provider calls.

The object resource resolves to `/2/files/upload`. The lifecycle resource resolves to `/2/files/list_folder/continue`.

## Push and Auth Validation

Dropbox GET challenge verification is exposed by getWebhookChallenge; POST callbacks can be validated with x-dropbox-signature HMAC-SHA256 over the raw body.

## Nango template

Nango anchor: `dropbox`; sync `dropbox-files`; read scopes `files.metadata.read` and `files.content.read`; writeback additionally requires `files.content.write`.

## Nango Fallback

Dropbox declares `dropbox-files` for scheduled fallback records using provider config key `dropbox`.

## ID Patterns

`/dropbox/{accountId}/files`: `^/.+$`.

`/dropbox/cursors`: `^[A-Za-z0-9._-]+$`.
