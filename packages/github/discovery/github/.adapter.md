# GitHub adapter

The GitHub adapter exposes repository pull requests, issues, reviews, comments, commits, files, and checks under `/github`, with writeback support for creating and closing pull requests, creating and updating refs, creating and updating issues, creating and updating issue comments, submitting pull request reviews, merging pull requests, and replying to pull request review comment threads.

Read-only mounts:
- `/github/repos/<owner>/<repo>/pulls/<pullNumber>/meta.json` - Pull request metadata.
- `/github/repos/<owner>/<repo>/pulls/<pullNumber>/files/<path>` - Pull request file records.
- `/github/repos/<owner>/<repo>/issues/<issueNumber>/meta.json` - Issue metadata.
- `/github/repos/<owner>/<repo>/issues/<issueNumber>/comments/<commentId>/meta.json` - Issue comment records (directory records so per-comment children such as reactions can nest without a file/dir collision).
- `/github/repos/<owner>/<repo>/commits/<sha>/metadata.json` - Commit metadata.

## Operations

| To... | Do... |
|---|---|
| Read | `cat <canonical-resource-path>` after listing the resource directory or following an alias when one is available. Use the resource table and ID patterns below to determine whether a resource uses a bare id, an adapter-specific slug/id filename, or an exact sidecar path such as `content.md`. |
| Edit | Write the resource update payload to the canonical resource path. For JSON resources, included mutable fields PATCH; fields marked `readOnly` in `.schema.json` are rejected. |
| Create | Write JSON to any non-canonical filename such as `create request.json`. The adapter creates the record at its canonical resource path and rewrites the draft as `{ "created": "<real-id>", "path": "<canonical-resource-path>", "url": "<provider-url>" }`. |
| Ignore | Editor scratch files named `partial.json`, `.tmp.json`, `.partial.json`, `*.tmp.json`, or `*.partial.json` are ignored and never treated as create drafts. |
| Delete | `rm <canonical-resource-path>` for canonical records. |

## ID Patterns
- `/github/repos/{owner}/{repo}/issues/<id>.json`: `^[1-9]\d*$`. Filenames that do not match this pattern are treated as create drafts.
- `/github/repos/{owner}/{repo}/issues/{issueNumber}/comments/<id>.json`: `^(?:meta|\d+)$`. Filenames that do not match this pattern are treated as create drafts.
- `/github/repos/{owner}/{repo}/pulls/{pullNumber}/reviews/<id>.json`: `^\d+$`. Filenames that do not match this pattern are treated as create drafts.
- `/github/repos/{owner}/{repo}/pull-requests/<id>.json`: `^[1-9]\d*$`. Filenames that do not match this pattern are treated as create drafts.
- `/github/repos/{owner}/{repo}/refs/<id>.json`: `^refs/[^/]+/[^/].*$`. The full normalized ref is percent-encoded as one filename segment; filenames that do not decode to this pattern are treated as create drafts.
- `/github/repos/{owner}/{repo}/pulls/{pullNumber}/close.json`: exact file path.
- `/github/repos/{owner}/{repo}/pulls/{pullNumber}/merge.json`: exact file path.
- `/github/repos/{owner}/{repo}/pulls/{pullNumber}/comments/{commentId}/replies/<id>.json`: `^\d+$`. Filenames that do not match this pattern are treated as create drafts.

## Write field contracts

### Create GitHub issue

Resource: `/github/repos/{owner}/{repo}/issues/<id>.json`
Schema: `/github/repos/{owner}/{repo}/issues/.schema.json`
Create example: `/github/repos/{owner}/{repo}/issues/.create.example.json`
Required fields: `title`.
Optional fields: `body`, `assignee`, `milestone`, `labels`, `assignees`, `issue_field_values`, `type`, `state`.

Fields:

- `title` (required, one of string, integer) - The title of the issue.
- `body` (optional, string) - The contents of the issue.
- `assignee` (optional, string or null) - Login for the user that this issue should be assigned to. _NOTE: Only users with push access can set the assignee for new issues. The assignee is silently dropped otherwise. **This field is closing down.**_
- `milestone` (optional, one of string, integer) - Milestone number or title.
- `labels` (optional, array) - Labels to associate with this issue. _NOTE: Only users with push access can set labels for new issues. Labels are silently dropped otherwise._
- `assignees` (optional, array) - Logins for Users to assign to this issue. _NOTE: Only users with push access can set assignees for new issues. Assignees are silently dropped otherwise._
- `issue_field_values` (optional, array) - An array of issue field values to set on this issue. Each field value must include the field ID and the value to set. Issue fields are only available for organization-owned repositories with the feature enabled. Field values are silently dropped otherwise.
- `issue_field_values[].field_id` (required, integer) - The ID of the issue field to set
- `issue_field_values[].value` (required, one of string, number) - The value to set for the field
- `type` (optional, string or null) - The name of the issue type to associate with this issue. _NOTE: Only users with push access can set the type for new issues. The type is silently dropped otherwise._
- `state` (optional, enum) - Issue state when updating an existing issue. Allowed values: `open`, `closed`.

### Submit GitHub pull request review

Resource: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/reviews/<id>.json`
Schema: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/reviews/.schema.json`
Create example: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/reviews/.create.example.json`
Required fields: `event`, `body`, `comments`.
Optional fields: `commit_id`, `metadata`.

Fields:

- `commit_id` (optional, string) - The SHA of the commit that needs a review. Not using the latest commit SHA may render your review comment outdated if a subsequent commit modifies the line you specify as the `position`. Defaults to the most recent commit in the pull request when you do not specify a value.
- `body` (required, string) - **Required** when using `REQUEST_CHANGES` or `COMMENT` for the `event` parameter. The body text of the pull request review.
- `event` (required, string) - The review action you want to perform. The review actions include: `APPROVE`, `REQUEST_CHANGES`, or `COMMENT`. By leaving this blank, you set the review action state to `PENDING`, which means you will need to [submit the pull request review](https://docs.github.com/rest/pulls/reviews#submit-a-review-for-a-pull-request) when you are ready. Allowed values: `APPROVE`, `REQUEST_CHANGES`, `COMMENT`.
- `comments` (required, array) - Use the following table to specify the location, destination, and contents of the draft review comment.
- `comments[].path` (required, string) - The relative path to the file that necessitates a review comment.
- `comments[].position` (optional, integer) - The position in the diff where you want to add a review comment. Note this value is not the same as the line number in the file. The `position` value equals the number of lines down from the first "@@" hunk header in the file you want to add a comment. The line just below the "@@" line is position 1, the next line is position 2, and so on. The position in the diff continues to increase through lines of whitespace and additional hunks until the beginning of a new file.
- `comments[].body` (required, string) - Text of the review comment.
- `comments[].line` (required, integer) - Line number on the selected side.
- `comments[].side` (optional, string) - Diff side for the comment. Defaults to RIGHT. Allowed values: `LEFT`, `RIGHT`.
- `comments[].start_line` (optional, integer) - First line for a multi-line comment.
- `comments[].start_side` (optional, string) - Diff side for the first line of a multi-line comment. Allowed values: `LEFT`, `RIGHT`.
- `comments[].suggestion` (optional, string) - Optional suggested replacement text appended to the comment body.
- `metadata` (optional, object) - Optional submission metadata.
- `metadata.commitSha` (optional, string) - Commit SHA to anchor the review to.
- `metadata.connectionId` (optional, string) - Relayfile connection id override.
- `metadata.providerConfigKey` (optional, string) - GitHub provider config key override.

### Create GitHub pull request

Resource: `/github/repos/{owner}/{repo}/pull-requests/<id>.json`
Schema: `/github/repos/{owner}/{repo}/pull-requests/.schema.json`
Create example: `/github/repos/{owner}/{repo}/pull-requests/.create.example.json`
Required fields: `title`, `head`, `base`.
Optional fields: `body`, `draft`, `maintainerCanModify`, `author`.

Fields:

- `title` (required, string) - Pull request title.
- `head` (required, string) - Head branch or `owner:branch`.
- `base` (required, string) - Base branch.
- `body` (optional, string) - Pull request description.
- `draft` (optional, boolean) - Whether to create the pull request as a draft.
- `maintainerCanModify` (optional, boolean) - Whether maintainers may modify the head branch.
- `author` (optional, enum) - Credential identity selected by the write orchestrator; it is not sent in the GitHub REST body. Allowed values: `app`, `user`.

### Push GitHub ref

Resource: `/github/repos/{owner}/{repo}/refs/<id>.json`
Schema: `/github/repos/{owner}/{repo}/refs/.schema.json`
Create example: `/github/repos/{owner}/{repo}/refs/.create.example.json`
Required fields: `ref`, `sha`.
Optional fields: `force`.

Fields:

- `ref` (required, string) - Full ref or branch name. For updates, it must match the ref encoded in the canonical filename.
- `sha` (required, string) - Git object SHA already present in the repository.
- `force` (optional, boolean) - Allow a non-fast-forward update when writing the canonical ref file.

### Close GitHub pull request

Resource: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/close.json`
Schema: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/close.json/.schema.json`
Create example: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/close.json/.create.example.json`
Required fields: none at the top level.
Optional fields: none.

Fields: none. Write an empty JSON object to close the pull request.

### Merge GitHub pull request

Resource: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/merge.json`
Schema: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/merge.json/.schema.json`
Create example: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/merge.json/.create.example.json`
Required fields: none at the top level.
Optional fields: `commit_title`, `commit_message`, `sha`, `merge_method`, `metadata`.

Fields:

- `commit_title` (optional, string) - Title for the automatic commit message.
- `commit_message` (optional, string) - Extra detail to append to automatic commit message.
- `sha` (optional, string) - Expected head SHA. GitHub refuses the merge if the pull request head has moved.
- `merge_method` (optional, string) - Merge strategy. When omitted, GitHub uses the repository default. Allowed values: `merge`, `squash`, `rebase`.
- `metadata` (optional, object) - Optional submission metadata.
- `metadata.connectionId` (optional, string) - Relayfile connection id override.
- `metadata.providerConfigKey` (optional, string) - GitHub provider config key override.

### Reply to GitHub pull request review comment

Resource: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/comments/{commentId}/replies/<id>.json`
Schema: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/comments/{commentId}/replies/.schema.json`
Create example: `/github/repos/{owner}/{repo}/pulls/{pullNumber}/comments/{commentId}/replies/.create.example.json`
Required fields: `body`.
Optional fields: none.

Fields:

- `body` (required, string) - The text of the review comment.
